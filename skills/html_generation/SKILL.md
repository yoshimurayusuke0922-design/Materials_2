# html_generation

## Purpose

構造化された情報から、編集しやすい HTML/CSS を生成する。

## Inputs

- `outputs/03_structure/` の設計ファイル
- `outputs/02_ocr/text_blocks.json`
- レイアウト解析結果

## Outputs

- `outputs/04_html_drafts/draft_v1.html`
- `outputs/04_html_drafts/draft_v2.html`
- `outputs/05_final/index.html`
- `outputs/05_final/style.css`

## Procedure

1. 見出しや本文を意味のある HTML タグへ落とす。
2. CSS で見た目を整える。
3. まず draft を作り、必要なら改稿する。
4. 最終版を `outputs/05_final/` に出力する。
5. レイアウト修正時は、局所的な `top` / `left` / `transform` / 個別 margin ではなく、親コンテナの grid / flex / gap / padding / width 配分を見直す。
6. カード上部でイラストと見出しを横並びにする場合は、header コンテナを作り、本文エリアとは分離する。
7. 中央揃えや右寄せなどの指定は、親だけでなく、実際にテキストやリストを持つ要素に当たっているか確認する。

## Quality Checks

- セマンティック HTML になっている
- PC 表示で読める
- スマホ表示で大きく崩れない
- 画像背景に頼りすぎていない
- 提案スライドのカード見出しは、手動 `<br>` に頼る前に印刷プレビュー幅で確認する。特に制度名、サービス名、括弧付き語句は「（一般型）」だけが単独行にならないよう、同一フレーズを1行に収めるか、意味単位で自然に改行する
- カード内のアイコンは、透明余白込みの画像サイズではなく見た目の実寸で確認する。印刷プレビューで小さく見える場合は、該当カード専用のCSSで表示サイズを大きめに取り、説明図として読める存在感を確保する
- イラスト周辺の余白が大きい場合は、画像サイズだけでなく画像コンテナの固定幅、padding、margin、gap、`object-fit` を確認する。透明背景のイラストは、重ならない範囲でテキスト側へ近づけてよい
- 左にデモ画面、右にカード群があるスライドでは、親を2カラムまたは3カラム構造として管理し、右寄せやデモ領域拡張は `grid-template-columns`、`justify-self`、`justify-content` で行う
- 赤線や装飾線より下だけを調整する場合は、上部header、本文段落、強調ボックス、箇条書きを別セレクタで扱う。本文は中央揃え、箇条書きは「ブロック全体を中央配置 + 内部左揃え」など、要素ごとの意図を明示する
- 数字など一文内の一部だけを強調する場合は、その部分を `span` に分け、フォントサイズ・行高・配置を局所的に調整する
- 日本語は単語、熟語、送り仮名、助詞・補助動詞だけが孤立しないよう、意味の切れ目で `<br>` を入れるか、テキスト幅を調整する

## Failure Cases

- 画像の見た目だけを追って div を乱立する
- テキストが画像化されて編集不能になる
- 余白や改行が不自然になる
- 括弧付き語句や助詞だけが孤立した改行になる
- 透明余白のあるアイコンを nominal size のまま置き、スライド上で小さく見える
- `text-align: center` を親にだけ指定し、実際の `p` / `strong` / `ul` / `li` に効いていない
- リスト全体を中央配置したい場面で、リスト内部まで中央揃えにして読みづらくする
- 右寄せや幅調整を `left`、`transform`、個別 margin で帳尻合わせする
- イラストの透明余白を画像サイズ変更だけで解決しようとし、本文領域が狭いまま残る

## Notes

- 完全一致より構造化・可読性・編集性を優先する
