# Componentized Image Pipeline

## 結論

- `1スライド = 1画像生成` はやめる
- `レイアウト固定 -> 部品分解 -> 部品ごとに画像生成 -> HTMLで合成`
  に切り替える

この方針の方が、HTML再現性、改修性、デザインの安定性が高い。

## なぜ一発生成が破綻するか

1. 画像生成モデルは全体整合性より局所の見た目を優先しやすい
2. 文字、装飾、図、余白を同時に制御しづらい
3. 後からHTMLで再現するとき、どこまでを画像にし、どこまでをテキストにするかが曖昧になる

## 新ルール

### 1. レイアウトを先に固定する

最初に以下を定義する。

- スライド種別
- 16:9内の大枠グリッド
- コンテンツ領域
- 余白ルール
- タイトル位置
- 本文位置
- 図やカードの占有領域

### 2. コンポーネントに分解する

スライドを次のような部品へ分ける。

- タイトル
- サブタイトル
- 本文
- カード
- アイコン
- 背景装飾
- 図解
- ロゴ
- スクリーンショット枠

### 3. テキストとビジュアルを分離する

HTMLで持つもの:

- 見出し
- 本文
- 箇条書き
- 注釈
- ラベル

image生成で持つもの:

- 背景装飾
- 図解の土台
- UIフレーム
- 抽象図形

### 4. 画像生成は部品単位で行う

NG:

- 表紙1枚をまるごと生成
- 提案の骨子1枚をまるごと生成

OK:

- 表紙の背景装飾だけ生成
- 4カードの共通カード土台だけ生成
- 展開イメージ中央ハブの図形だけ生成
- デモ画面枠だけ生成

### 5. HTMLで組み立てる

- レイアウトとテキストはHTML
- 装飾と図は必要箇所だけ画像差し込み
- 同じカードや枠はCSSクラスで再利用

## 今回の資料に当てはめた分解例

## 表紙

### HTMLで持つ

- 宛先
- PROPOSAL
- タイトル
- サブタイトル
- 日付

### 画像またはCSSで持つ

- 右の赤い有機形状
- 上部の淡い円形ニュアンス
- 左下の赤い形状
- ロゴ台座の雰囲気

## 提案の骨子

### HTMLで持つ

- 見出し
- リード
- 4カードのタイトル
- 4カードの説明
- 下部メッセージ

### 画像またはCSSで持つ

- カード共通装飾
- 赤いアクセントバー
- 必要なら小アイコン

## デモ画面イメージ

### HTMLで持つ

- セクション見出し
- キャプション
- ラベル

### 画像またはCSSで持つ

- UIフレーム
- 空のスクリーンショット枠
- 画面群のベースデザイン

## 展開イメージ

### HTMLで持つ

- 各カード見出し
- 説明文
- 中央ラベル

### 画像またはCSSで持つ

- 中央ハブ図形
- 接続線
- 周辺ノードの飾り

## 実行順

1. `component_layout_planning`
2. `component_image_generation`
3. `component_html_assembly`
4. `quality_review`

## 判断基準

優先順位は次のとおり。

1. 情報が正しく伝わる
2. HTMLで編集しやすい
3. デザインの統一感がある
4. 再利用できる
5. 元画像にある程度近い
